### Step-3 spike output (Plan 005) — completed 2026-06-10

The ws-server listens with `httpServer.listen(port, ...)` (no host arg) → Node binds all
interfaces, so the ws port is reachable on the tailnet. After the hostname fix, per topology:

1. **Laptop Electron → Mac Mini over raw tailnet (no Serve): WORKS.** The renderer now dials
   `ws://<mac-mini-tailnet-host>:<wsPort>/events`, which the all-interfaces ws-server accepts
   directly over the tailnet. This is the case the operator described ("open in Hilt" from a
   Mac Mini agent → laptop renderer) and it is fixed by Plan 005 alone.
2. **iPhone PWA → Mac Mini over Tailscale Serve (https): NEEDS A FOLLOW-UP PLAN.** Tailscale
   Serve proxies a single origin (443 → the Next app on 3000). The ws-server runs on a
   *separate* port that Serve does not proxy, so `wss://<serve-host>:<wsPort>/events` is not
   reachable even with the correct hostname. The hostname fix is necessary but not sufficient
   here. **Recommended follow-up**: serve the `/events` WebSocket from the same origin as the
   Next app (host the HTTP `upgrade` on the Next origin; bind the internal ws-server to
   localhost), so the gateway's single `/hilt → :3000` route carries it. **Now folded into the
   internal app gateway plan** at `work/meta/internal-app-gateway-plan.md` ("Real-Time Events
   (WebSocket) Over the Gateway"), to be built with that plan's base-path + URL-helper work —
   not as a standalone plan in this batch.

## What this batch deliberately does NOT include

The operator reviewed the full audit and scoped this batch. Held for later (not rejected):

- **Library `rehypeRaw` XSS hardening** (`src/components/library/LibraryMarkdown.tsx:125`) — external/ingested content is rendered through `rehype-raw` with no sanitizer. Real and relevant (ingested feeds are untrusted regardless of network), but more involved than this batch (must add sanitization without breaking rendered media/embeds). Pick up as a fast follow.
- **Broad origin/host guard for the 0.0.0.0-bound server** — lower urgency on a trusted tailnet, and any guard must NOT break legitimate tailnet/phone access (the operator relies on it). Plan 002 removes the one sharp edge (the `reveal` RCE); the systemic guard is a separate, carefully-scoped effort.
- **Dependency bumps** — runtime advisories worth a small bump: `ws` (moderate, used in the ws-server) and `yaml` (moderate, parses local config). `uuid` moderate affects only v3/v5/v6 with a caller-supplied buffer (Hilt uses v4) → low real risk. The headline `npm audit` count (~37, incl. "critical") is ~90% dev/build tooling (electron-builder chain, `concurrently`) that never ships in the running app. Fold the two runtime bumps into the 001 branch when convenient.
- **CI workflow** — intentionally deferred; this is a local-first single-developer repo (see Plan 001 maintenance notes).
- **Compiled `electron/main.js` committed alongside `electron/main.ts`** — minor; gitignore + rebuild-on-launch is a trivial future cleanup, not worth a plan.

## Findings considered and rejected (do not re-audit)

These were reported by audit subagents and **rejected during vetting** — the cited code was opened and the claim did not hold. Recorded so future runs don't resurface them:

- **Calendar dedup "winner gets hidden"** (`src/lib/calendar/db.ts:238-253`): FALSE. `refreshVisibleDuplicates` iterates each row exactly once (`for (const row of rows)`); the winner is shown on its own iteration and never re-hidden. The subagent imagined a nested double-iteration that does not exist. Code is correct.
- **`getCalendarById(id)!` crash** (`src/lib/calendar/db.ts:189`): BY-DESIGN. The non-null assertion follows a synchronous `INSERT … ON CONFLICT DO UPDATE` immediately read back on the same better-sqlite3 connection; the row is guaranteed present. Not a bug.
- **Firecrawl SSRF** (`src/app/api/firecrawl/route.ts`): FALSE. Hilt only ever `fetch`es `https://api.firecrawl.dev/v1/scrape`; Firecrawl's hosted service fetches the user URL, not Hilt's server. No `file://`/metadata SSRF against Hilt's network. (At most a remote peer could waste API credits — not the reported finding.)
- **osascript folder-picker injection** (`src/app/api/folders/route.ts:140`): NOT A FINDING. The AppleScript is static and hardcoded; no user input is interpolated.
- **MCP server "missing"** (direction): REJECTED by the operator as not needed. Hilt is a display surface; agents run on the same machine as the vault and act on files directly, so an MCP server exposing Hilt would be redundant. The genuinely valuable cross-machine gap (an agent telling the viewer's screen to navigate) is addressed by Plan 005, not by MCP.
- **Agent write-surface asymmetry** (direction): NOT A PROBLEM for this deployment. Agents are co-located with the vault and write files directly, which round-trips through Hilt's file-native model by design. A REST write-API matters only if a client with no local filesystem access (e.g. the phone) must mutate the vault without an agent — deferred until that's a real need.
