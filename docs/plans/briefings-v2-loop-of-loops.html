North star. Independent domain loops — each producing content, escalations, and a self-evaluation — with one daily briefing that reads them. The eval dimension exists so each loop's stability and quality improve as it runs. In this ecosystem an agent is a loop plus action-taking consumers — same organism, two growth stages. This pattern is intended to become a fundamental backbone of Hilt and the wider personal agentic ecosystem: adding a new function = adding a loop that satisfies the contract; everything downstream (briefing surface, health watch, verdict system) picks it up automatically.

1 · Intent

Redesign the briefing from a single daily synthesis into a loop-of-loops report. Named gaps in the current system:

  • "Don't drop this" has no memory — it's model judgment re-derived each morning over a sliding window (meetings last 4 days, newest task list, last 2 briefings as its only dedupe memory), not a durable action ledger. A commitment from 5+ days ago that never became a task is invisible.
  • Meeting notes are excerpted but never reconciled against projects, code, roadmap docs, or completed work.
  • "Work & product" promises project/roadmap status, but the gatherer supplies no project or roadmap documents at all — only git logs (2 days, 5 repos) and session filenames. The model derives "roadmap status" from commit messages and meeting excerpts.
  • Design principle (keep it): avoid hard-coding permanent briefing rules; loops produce evidence, the synthesis layer keeps all judgment — the MAINTAINING.md philosophy, extended from two layers to three: loops → gather → synthesis.

Cadence: there is no separate weekly analysis. The daily briefing is the only reader; it reads whatever loop artifacts are fresh. The existing weekend edition stays what it is — a daily briefing spanning Sat–Sun because those are low-activity days. Cadence is a property of the loop, not the briefing: if analysis feels redundant day after day, slow that loop's cadence; don't fork the briefing into editions.

The morning briefing is the right shape for two reasons: it's the right analysis interval (loops get the night to run), and it matches the human bandwidth — Justin can critique, evaluate, and improve this system at most once a day. Both the content and the pending verdicts therefore batch into the morning.

2 · The pattern is already proven

The library is the existence proof — and it proves the full pattern. It already has every dimension of a proper loop:

  • Content: the 05:10 morning report (bridge/meta/library-reports/YYYY-MM-DD.md) — substance, scorecard, deltas — plus the weekly editor's memo.
  • Self-eval: the steering loop — quality scorecard, judge-vs-formula disagreement, and proposals for tuning its own pipeline, escalated for Justin's verdict.
  • Granularity below the loop: every library item runs its own mini-loop (capture → digest → reweave → connections), and carries its own per-item feedback (comments, eval critiques, review lanes for pipeline versions). The library is fractal — a loop of item-loops. This depth is the special sauce to preserve, and the reference for how other loops may deepen later (§6).

That proposals-under-supervision flywheel is the thing to generalize. v2 = replicate this shape for the other domains, then thin the briefing to a reader.

Smell worth fixing — briefing SKILL.md §4
The briefing skill (bridge/meta/skills/briefing/SKILL.md — the prose the synthesis model reads every morning) has drifted into hard-coded per-loop rendering policy in its Library section: "ALWAYS end with [Full library report](/api/reports/morning)", memo-thesis handling, exact link formats. Not bad judgment — bad placement: library rendering rules living in the global skill, because the library is the only loop and its rules had nowhere else to go. With the contract, §4 reduces to one generic rule ("per loop artifact: lead with its escalations, headline notable deltas, link the full artifact") plus a bounded few lines of per-loop prose where a loop genuinely needs it (decided in interview — judgment stays in prose, per MAINTAINING.md; the briefing loop's health pass watches for rule-creep).

3 · The ontology & the loop contract

Highest-leverage decision; define first. The data model everything else hangs off:

3.1 · The ontology

Every loop reads its sources and emits items — the insight is the atom. Three kinds:

ItemWhat it isWants from Justin
InsightAn observation. "Task X is 4 days overdue." "Tomorrow's agenda is missing pricing." Inert — it informs.Nothing (feedback always possible)
ActionProposed work, executable by human or agent, with enough context to be picked up by either. Lifecycle: proposed → approved → in-progress → done.A verdict
ProposalProposed change to the system itself ("stop surfacing calendar items younger than 2 days"). The meta-twin of an action.A verdict

Properties any item carries: provenance/citation (always); escalated: true + reason (set by the loop at emit time — meaning "top fold today"); confidence (actions/proposals).

Escalation is urgency, not a type and not a separate pass. Any item kind can be escalated, and escalated-ness is orthogonal to needing a verdict: "big presentation tomorrow, get some sleep" is an escalated insight — top-fold worthy, nothing to approve. A low-confidence ledger action sitting quietly in a review queue is the reverse — not escalated, still wants a verdict eventually.

Responses — how Justin's judgment comes back — are two, and they attach at every level (item, section, the briefing itself — including absence feedback: "I expected an insight about the Seb meeting here" lands at section level):

  • Feedback — free-form, attaches to anything, never blocks, accumulates until a health pass consumes it.
  • Verdict — an enumerated choice on an item that asked (actions & proposals only): approve / dismiss / assign-to-me / assign-to-agent / revise-with-note. Executes mechanically — no interpretation needed. Counted (hit rates). revise-with-note is deliberately hybrid: the revise is a verdict, the note is feedback embedded in it.

Layers — work vs. meta is what an item is about; feedback vs. verdict is how you respond. All four cells are real:

Work layer (the domain)Meta layer (the system)
Verdict"Yes, do this task" / "assign to me""Bless this tuning proposal"
Feedback"This action is scoped badly" / "you missed the real commitment""This shouldn't surface here" / "too verbose"

The layers are structurally symmetric: the meta layer's insights are health findings; its actions are tuning proposals. The health pass is just a loop whose sources are feedback + run data, emitting meta-items the way domain loops emit work-items. Two nested cycles fall out:

WORK CYCLE (daily)      loop reads sources → emits items → briefing renders
                        → verdicts on actions → work happens → state closes

IMPROVEMENT CYCLE       feedback accumulates → health pass interprets
(slower, per loop)      → meta insights + tuning proposals → verdicts → loop changes
                        → the change is measured

3.2 · The artifact contract

Every loop runs on its own cadence and writes a durable, dated artifact. The contract requires two structured sections + frontmatter; the content dimension is loop-defined:

---
loop: meeting-actions
run_at: 2026-07-06T05:10:00Z
cadence: daily
---
## <content sections — loop-defined>   ── CONTENT: the loop's items (insights + actions),
                     shaped per domain. Scorecard/Deltas is the recommended convention
                     where it fits (library, meetings, runtime); loops with different
                     shapes (calendar, people projections) define their own sections.

## Escalations    ── REQUIRED, but a VIEW not a type: the loop's items marked
                     escalated, collected for the reader. Each item is STRUCTURED —
                     stable id (minted by the loop), kind, provenance, and for asks the
                     allowed verdicts — so the UI can render and act on it. Most days empty.

## Loop health    ── REQUIRED: the loop's self-eval, run alongside every execution:
                     run metrics (attempted/succeeded/coverage),
                     quality eval (is the output good; drift vs. recent runs),
                     proposals (tuning changes awaiting verdict — steering, generalized)

Consequences:

  • Uniformity lives exactly where the machinery needs it — escalated items feed the top fold; asks feed the verdict system; health feeds the runtime loop — and freedom lives where domains genuinely differ.
  • Do not build a separate escalation-queue system. The queue is the union of ## Escalations views; the briefing is its reader.
  • Retrofit the library (morning report + steering, as one loop artifact) to the contract as loop #1 — cheap, validates the shape against the loop that already works.

4 · Source vs. rendering: the briefing is a UI

There are two kinds of artifact, and keeping them distinct is load-bearing:

  • Loop artifacts — the durable source of truth, written by loops on their own cadence.
  • The briefing — a rendering of the fresh artifacts, sized to the morning read. It holds no data of its own.

The briefing render: top fold = the union of every loop's escalated items (this is where "Don't drop this" survives — as a UI mechanism, not a section with its own sourcing), then one section per loop = its insights and actions as headlines + a health glyph (health surfaces only on failure, drift, or a pending proposal). All judgment stays in synthesis prose; what to remember is no longer judgment (§6, dedupe).

Because items are structured (ids, kinds, allowed verdicts on asks), Hilt's briefing view can render asks as actionable — approve / dismiss / assign / revise — not paragraphs. And every rendered bullet, ask or not, carries a feedback affordance (§6): critique attaches to any item, any section, or the briefing itself, and routes to the owning loop. The markdown briefing file stays the portable rendering; the Hilt UI on top of it evolves freely (the top fold, per-loop sections, and feedback/verdict affordances are UI concerns, deliberately kept out of the loop inventory).

gather.sh (the deterministic pre-synthesis data collector — pure bash, no judgment, prints raw material into the model's context) stops collecting raw domain data as loops come online: it reads the registry (§5) and prints each registered loop's latest artifact. Adding a loop requires zero gather changes.

Missing-artifact semantics: if a registered loop's artifact is missing or stale, the briefing renders that section as "loop did not run" (sourced from the runtime loop's report) — it never fabricates the section from raw data. Scheduling & budget: loops are claude -p runs; they stagger overnight and must finish before gather (~06:00). Rate limits are a proven failure mode (the reweave outage) — the degradation policy is: a loop that can't run skips cleanly, the runtime loop reports the absence, and the briefing says so.

5 · Registry, artifact homes, and retention

The filesystem encodes the source/rendering split (§4). Root-level folders (/meetings, /references, /briefings, /people, /areas, /lists) are content for consumption; /meta is machinery. An agent perusing the vault should instinctively know which is which — so loop artifacts, ledgers, feedback, and verdicts never live in content space. (This supersedes an earlier "domain-co-located" decision, which would have put machine reports inside content folders; it also avoids the previously-rejected central bridge/reports/ dump in content space.)

Convention: meta/loops/<domain>/, mirroring root domain names, with a uniform inner shape — the contract made visible in the directory listing:

meta/loops/
  registry.yml              # loop → cadence, artifact globs, state files — the single lookup
  references/               # mirrors /references (the library loop)
    reports/YYYY-MM-DD.md   #   dated artifacts — morning report + memo migrate here at conformance
    state/                  #   scoring, muted senders, review queues (today flat at meta/ root)
    feedback/  verdicts/
  meetings/                 # mirrors /meetings
    reports/YYYY-MM-DD.md
    state/ledger.md         #   THE action ledger
  people/                   # mirrors /people — home follows DOMAIN, not generator
    <slug>.md               #   person/group projections, written by the meeting loop
                            #   (initially — registry records the writer; if people ever
                            #   graduates to its own loop, nothing moves)
  briefings/                # the renderer-loop's own machinery
    state/surfacing.json    #   escalation dedupe memory
  system/                   # runtime loop (no root mirror — its domain IS the machine)
  areas/                    # goals loop

Mirror domain names, not loop names — the registry maps loop → home, and discoverability comes free. Registry consumers: gather (print latest artifacts), the runtime loop (absence detection), and Hilt (rendering). Existing migrations at conformance time: meta/library-reports/ → meta/loops/references/reports/ (the /api/reports/* routes re-point); references/process/memos/ → same reports home (the memo is a loop artifact; it reaches Justin through the briefing + Library rendering).

Published surfaces stay in content space: /briefings (the daily rendering), the People view (composes the human-authored people/<name>.md note + the live projection from meta/loops/people/ at render time — person notes are never machine-churned, and no dated person snapshots: history reconstructs from the ledger + dated reports). If a person page ever earns durable synthesized prose, that graduates to content space — published = content tree, machinery = meta, always.

Retention: dated files, forever. Text is cheap; history is how the health dimension measures drift and how future derived loops read backwards. .archive/ sweeps if a directory ever gets unwieldy. The registry always knows how to find "latest." Ledgers are state, not reports — they live outside the dated artifacts and persist independently. Commit discipline: each loop commits its own artifacts at run end (per-loop commits, the briefing generator's existing pattern).

6 · The verdict & feedback system

A first-class element of this scope: the mechanism by which Justin's judgment flows back into loops — and the loops' hit rates earn autonomy. The two response types from the ontology (§3.1), operationalized:

The rule
Every rendered item is feedback-able; asks (actions & proposals) additionally take a verdict. Escalation is urgency, not ask-ness — an escalated insight needs no verdict; an un-escalated action still wants one.
  • Feedback — free-form editorial critique attachable to any item, section, or the briefing as a whole: "redundant," "wrong emphasis," "cited the wrong meeting," "more like this," "I expected an insight here." Nothing asked a question; nothing is blocked. Always available — exactly as every library item can receive comments regardless of whether it asked for anything.
  • Verdicts — a selection from outcomes the loop enumerated in advance, recorded as a state transition the loop executes mechanically, and countable as a measure of the loop's judgment. Attaches only to asks. Work-cycle verdicts decide work ("do this task"); improvement-cycle verdicts decide system changes ("bless this proposal") — same response type, both counted.

The flywheel (why feedback must be universal)

The library already runs the full circuit, and it is the thing being generalized:

  1. Feedback on any item → stored, routed to the owning loop
  2. The loop's health pass processes unprocessed feedback — diagnose, cluster into patterns, propose fixes (this is the existing process-library-feedback flow, generalized)
  3. Proposals surface as escalations (tuning_proposal)
  4. A verdict gates the change → the loop changes → the change is measured

Feedback is the input to the improvement loop; verdicts are the gate on it. Without universal feedback, the health dimension has no human signal and the flywheel spins only on self-assessment.

What we build

  • Feedback capture on every rendered item — a comment affordance on any briefing bullet (the library's per-item comment pattern, promoted to universal). Anchoring: escalations anchor by loop-minted id; content bullets anchor by (briefing date, section/loop, bullet text + citation) — every bullet already carries a citation per the existing SKILL.md rule, and the owning loop is derivable from the section.
  • Feedback routing — each item's feedback lands with its owning loop, consumed by that loop's health pass on its next run. Feedback about the rendering itself ("top fold ordering," "too verbose") has an owner too: the briefing generator registers as a loop under the same contract — its domain is rendering the other loops, its health dimension is render quality, and critique of the briefing is its feedback stream (MAINTAINING.md's "weekly review" is the manual version of this today).
  • Item schema — id (stable, minted by the emitting loop), loop, kind (insight | action | proposal), escalated + reason, confidence, payload, and — for asks — allowed verdicts (approve / dismiss / assign-to-me / assign-to-agent / revise-with-note).
  • Two UI affordances — feedback (comment) on every bullet; verdict buttons additionally on escalations — wherever items render: briefing view, People view, Library.
  • Verdict log — per-loop, at meta/loops/<domain>/verdicts/ (registry-discoverable): each loop reads its own on its next run: act on approvals, drop dismissals, incorporate revision notes; and the substrate for hit-rate measurement, which is what graduates a loop toward auto-accept (each loop's health dimension tracks its own approve/dismiss outcomes).
  • author on every feedback/verdict record — justin | claude-sim: the launchpad (§10) exercises the real capture paths with simulated engagement, and that signal must never pollute real feedback or hit-rate stats.
  • Surfacing state (escalation dedupe) — the briefing keeps a small read-state (id → first_surfaced, times_surfaced, verdict), analogous to the library's read-state. Aging is computed: first appearance = full headline; unresolved repeat = one terse line with age ("day 4"); verdict recorded = suppressed. The model keeps editorial latitude over what the state suggests (it may resurface or expand when newly relevant) — but overrides are visible: the briefing notes when it surfaces something the state would have suppressed. Memory is deterministic; judgment stays with the model.

The library's role: first instance, not exception

Under the corrected model, the library's per-item feedback machinery isn't domain-specific depth we carve an exception for — it's the first instance of the universal rule (every surfaced item is feedback-able), plus the reference implementation of the flywheel (library-feedback → process-library-feedback → proposals → review lanes). The unification rule stands, reframed:

  • Migrate the loop-level layer: steering proposals adopt the shared schema + verdict log (the prototype for tuning_proposal; the existing review-lane flow becomes the first consumer of the unified surface rather than a parallel system).
  • Keep the item-level machinery library-native for now: per-item comments, eval feedback, and pipeline-version review lanes keep working exactly as they do — they already implement the universal rule for their domain. Converge storage/schema opportunistically, never at the cost of the granularity.
  • Reuse the library's shape as other loops deepen: ledger entries are the obvious analog (per-entry feedback + verdicts is already item-level); the meeting loop's feedback processing should reuse the process-library-feedback pattern rather than invent one.

Unify the interface; preserve the depth.

7 · The runtime loop (system health)

Per-loop self-eval cannot cover two things — exactly what a general system-health report is for:

  1. Absence detection. A dead loop writes no artifact — self-reporting can't report a run that never happened. The runtime loop reads the registry and flags stale/missing artifacts.
  2. The substrate. Things that aren't loops: machines/disk/sync, the app supervisor, the schedulers themselves, credentials and API budgets (the Claude OAuth token, the Gemini key, the remindctl TCC permission — all real failures from recent weeks), backups.

It also highlights notable items from the per-loop health dimensions — the cross-loop health digest — without owning them. Kernel-to-processes analogy: loops self-report their quality; the runtime loop reports their existence and the ground they run on.

Case study — why absence detection is load-bearing
2026-06-29 session: the nightly library reweave failed silently for weeks — Claude preferred a stale Keychain token over the OAuth token under launchd, every scheduled call 401'd, and 47 items backlogged before a user complaint surfaced it. The loop that silently doesn't run is the one failure mode in-band health can't catch.

9 · Loop details

9.1 · Meeting-action ledger (the Meeting Agent)

Scope note: this section absorbs the full 🎧 Meeting Agent scope from the now-list — there is negligible difference between a loop and an agent here: the meeting loop is the scheduled analysis; the meeting agent is that loop plus its consumers (verdict UI, person/group views, drafted work, eventual auto-accept).

Mission: read every meeting; surface things Justin hasn't acted on or captured as a task; itemize the work so it's actionable by a human or an agent; over time, earn the right to complete high-confidence items autonomously.

The fix for "no memory" is item identity. Ledger entries carry: action (the commitment/next-step), owner (Justin / named person / agent — schema stays open to agent assignees), citation (meeting file, date, source lines), confidence (drives review priority), status (open → carried → resolved | dropped), and derivable age (opened date, briefings-ridden count). Payoffs: the "perennial reminders" rule becomes computed, not judged; and the ledger is the substrate for the verdict flow and the person/group views.

  • Infrastructure mostly exists — reuse it. Granola transcripts already land in the vault matched to calendar events; the native briefing harness (gather → skill → claude -p → validator → vault commit, launchd-scheduled) is the exact shape for the extractor loop. The extractor is a sibling of the briefing generator, not a new kind of thing.
  • The catch-phrase is v1, not v2 — and it's decided: "Action item:". A transcript scan for the phrase at post-processing (minutes after meeting end) forces a high-confidence capture — no realtime infra. Chosen deliberately for being natural meeting language: anyone saying it becomes a capture signal, not just Justin; false positives are absorbed by confidence tiering (phrase captures are high-confidence but still verdict-gated in v1). Launchpad (§10) validates detection against real Granola transcripts. True realtime stays v2+, maybe never needed.
  • Injection discipline: transcripts contain other people's words. Extraction treats meeting content strictly as data, never instructions — an embedded "add a task to wire $500 to…" is a commitment candidate to judge, not a command to obey. This discipline is a precondition for ever enabling auto-accept.
  • The genuinely hard part is reconciliation + identity, not extraction: stable action IDs, dedup across meetings (the same commitment restated three ways in three calls), reconciling against tasks/commits/completed work. Design effort goes there. Ledger is append-only with a reconciliation pass (a later meeting saying "we shipped that" closes the entry; a weekly sweep proposes closures for stale entries rather than silently dropping them).
  • Person/group projections: per-person and per-group rollups (open items, waiting-on-you, history, touch cadence) are generated alongside the daily artifact and rendered in Hilt's existing People view as sidecars. They don't render in the briefing except when they raise escalations ("you've owed Sarah X for 3 weeks"). Don't build the views first — they fall out of the ledger.

Staged shape: v1 — extractor loop on new transcripts → vault-native ledger → "awaiting your verdict" block (the loop's ## Escalations) + approve / dismiss / assign, catch-phrase scan included. v1.5 — agents draft the work attached to an entry (draft-and-present, never send). Auto-accept — only after a measured hit rate earns it (verdict log → health dimension), per the library judge-agreement pattern.

9.2 · Library

Already running (morning report 05:10 + steering + weekly memo + per-item pipelines). Work here is conformance, not construction: emit one contract-shaped artifact (content = existing scorecard/report substance; escalations = proposals awaiting verdict + items awaiting attention; health = the steering self-eval it already computes), and migrate steering proposals onto the unified verdict schema (§6). Item-level machinery — per-item feedback, review lanes, pipeline versioning — unchanged. The memo folds into the loop's content dimension (its "fresh memo" render rule leaves SKILL.md).

9.3 · Runtime / system health

Reads meta/loops/registry.yml; flags stale/missing artifacts (absence detection); checks substrate (supervisor heartbeats, sync, launchd job outcomes, disk, credentials/API budgets/TCC); digests notable per-loop health items. First loop to build after the contract — cheap, exercises the whole pipeline, and would have caught the reweave outage.

9.4 · Goals/areas alignment

Accountability loop: what we said matters (North Stars + area goals in areas/) vs. where attention actually went (ledger actions, commits, meetings, library saves, calendar time). A derived loop — reads other loops' artifacts plus git/calendar, barely touches raw vault — so it's cheap and it naturally absorbs the "drift" concern's v1: stale priorities and unkept directional promises surface here, with evidence. Escalates contradictions ("Product Factory is a top North Star; zero attention in 12 days") rather than enumerating every goal — the SKILL.md rule that goals are a relevance prior, not a checklist, carries over.

9.5 · Calendar/tasks

Already deterministic in gather (calendar today+tomorrow, reminders, newest weekly list with due dates). Conform to the contract last, for uniformity — its escalations (overdue, deadline-today) join the top fold; its health dimension finally gives the remindctl-hangs-under-launchd failure a place to surface.

10 · Launch evaluation — the launchpad

The ship gate is not "I built it." The plumbing will work on first launch; the judgment — extraction quality, escalation thresholds, phrasing — will be wrong in specific ways, which is exactly what the improvement flywheel exists for. So the bar is: the system has been run through its paces against history, graded against reality, iterated, and the builder can say "I've watched it be wrong, fixed it, and here's the evidence it converged."

The vault is a complete backtest environment: meetings back to Oct 2024, weekly lists per week, git history — and the actual briefings that shipped on those mornings. We can generate retro briefings and grade them against both the real briefing and what subsequently happened.

The harness

  • --as-of mode on every loop (a contract requirement): read only data ≤ the given date; write artifacts to a sandbox, never the live vault (generalize the briefing generator's existing --shadow infra).
  • Retro sweep: one representative day per week, ~8 weeks back — full stack per day (loops → artifacts → briefing).
  • Three-way grading per day: (1) vs. source data — every item cited and true, no fabrication; (2) vs. the actual briefing that shipped that morning — what did each catch or miss; (3) vs. subsequent reality — did escalations correspond to things that actually needed attention; did the ledger's open items match what later got done, dropped, or slipped.
  • Gold-labeled extraction set: ~30 historical meetings, commitments hand-labeled (careful pass), extractor measured on precision/recall. The identity test: the same commitment restated across meetings must yield ONE ledger entry. The reconciliation test: commitments provably completed later (commit/list evidence) must get closed when the pass runs across the historical sequence.
  • Simulated engagement — the flywheel's integration test: the builder reviews generated briefings as first-pass critic, submits feedback through the real capture path (as author: claude-sim, §6), verifies the health pass consumes it and produces sane proposals, and issues verdicts to prove the round-trip end to end — before Justin ever touches it.
  • Calibration boundary (per Justin): during the launchpad the builder optimizes on general, rationally-defensible quality principles — accuracy, citation integrity, no fabrication, sane escalation — and deliberately does NOT over-fit to Justin's personal nuance from historical data, which is always a facsimile of reality. The subjective fine-tuning (80% → 98%) happens in live use, through the feedback system, in the first week-plus.
  • Shadow week = the on-ramp of the ongoing review process, not just a gate: v2 generates alongside the live v1 briefing, and Justin compares them in an easy side-by-side / toggle UI in Hilt (a v1 build requirement of the briefing view), giving heavy feedback through the new system itself. Cutover when the comparison consistently favors v2.

The launch report (ship gate)

A written artifact, reviewed by Justin before cutover:

  • Retro coverage + per-day grades; extraction precision/recall vs. the gold set; identity/reconciliation results
  • Flywheel round-trip evidence (feedback → health pass → proposal → verdict → change)
  • An honest couldn't-fetch / couldn't-parse inventory — every data source the loops were supposed to read but couldn't, with cause
  • Interface walkthrough — is the briefing + verdict/feedback surface usable and legible as built
  • The builder's verdict: "here's why I believe in it" — or "here's why I don't yet," with the list of what's blocking

11 · Sequencing

  1. Land the native-briefing cutover (shadow → live via briefing:scheduler:install -- --variant live). Don't fork the skill across Hermes + Hilt mid-redesign.
  2. Contract + registry + verdict schema — the three shared primitives, defined together (short spec + a reader/validator in src/lib/briefing/). Retrofit the library as loop #1 to validate.
  3. Runtime loop — registry-driven absence detection + substrate. Exercises everything end-to-end.
  4. Launchpad harness — --as-of + sandbox writes + retro generation + grading scripts (§10). Built BEFORE the meeting loop, so the ledger is developed against the gold set from day one.
  5. Meeting-action ledger v1 — extractor → ledger → verdict block, catch-phrase scan included. Before any Meeting Agent views/execution.
  6. Goals/areas loop — derived; lands once ledger data exists to read.
  7. Thin SKILL.md to the generic per-loop rendering rule; top fold replaces "Don't drop this" sourcing; calendar/tasks conforms. v1.5 (drafted work) and person/group projections follow on the ledger.
  8. Shadow week → launch report → cutover (§10). No step ships to the live briefing without passing through the shadow.

Each step ships alone; the briefing improves monotonically.

Closed by interview (2026-07-02)

  • Budget: adaptive. Quality-first tiers during the launchpad (Opus-class for judgment-heavy loops), measure quality-per-token, then tune each loop to the cheapest tier that holds its grades. The registry records each loop's envelope.
  • Commits: per-loop. Each loop commits its own artifacts at run end (the briefing generator's existing pattern) — fine-grained provenance; the runtime loop can read git history as a health signal.
  • Catch-phrase: "Action item:" is the trigger — natural meeting language, and deliberately so: anyone saying it in a meeting becomes a capture signal, not just Justin. False-positive tolerance is handled by confidence tiering (phrase-triggered captures are high-confidence but still verdict-gated in v1). Launchpad validates detection against real Granola transcripts.
  • Verdict log: per-loop — meta/loops/<domain>/verdicts/, registry-discoverable, matching the uniform inner shape; hit-rate stats naturally scoped.
  • Content-bullet anchoring: citation anchor, validated in the launchpad — escalate to loop-minted content ids only if rewording breaks threads in practice. Evidence before machinery.
  • Render hints: small per-loop prose in SKILL.md — judgment lives in prose (the MAINTAINING.md philosophy, chosen over structural purity). Guardrail against the §4 rot: per-loop prose is bounded (a few lines each), and the briefing loop's health pass watches for rule-creep and flags it as a tuning proposal.
  • Escalation dedupe: model keeps latitude. The surfacing state informs (ages, counts, verdicts) but the model may override — resurface a suppressed item it deems newly relevant, expand a terse repeat. Guardrail: overrides are visible — the briefing notes when it resurfaces something the state would have suppressed, so the no-memory failure mode can't sneak back silently.
